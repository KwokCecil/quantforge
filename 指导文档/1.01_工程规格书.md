## 一、项目概述

### 1.1 项目定位

**QuantForge** — 量化研究与实盘部署的精简工程框架。

核心目标：

1. **策略无关性**：框架不预设策略逻辑，只提供策略落地所需的抽象接口
2. **回测-实盘一致性**：同一策略代码在回测和实盘中运行，区别仅在数据源和执行器
3. **统计验证**：内置因子检验和 Walk-Forward 验证基础设施
4. **多策略并存**：多个独立策略可并行运行，各自管理资金池和持仓

### 1.2 框架解决的核心问题

| 问题 | 解决方案 |
|---|---|
| 策略代码混合了数据获取、指标计算、交易执行、通知推送 | 分层架构：策略只产出 Decision，其余由框架完成 |
| 每个策略各自拉数据、格式不统一 | 统一 DataFeed 抽象，多源异构数据标准化接入 |
| 回测代码与实盘代码完全不同 | 同一策略代码，DataFeed/Executor 依赖注入切换 |
| 策略未经统计验证直接实盘 | 内置 FactorLab + Validator |
| 参数硬编码，无法扫描优化 | StrategyConfig 数据类 + 参数扫描基础设施 |
| API token 等硬编码在源码中 | 环境变量 + .env 文件 |

### 1.3 技术栈

|---|---|---|

## 二、系统架构

### 2.1 分层架构

```
┌──────────────────────────────────────────────┐
│              调度层 Scheduler                  │
│  回测调度 / 盘中监控调度 / 多策略并行           │
├──────────────────────────────────────────────┤
│              策略层 Strategy                   │
│  策略只产出 Decision，不执行交易                │
├──────────────────────────────────────────────┤
│              组合层 Resolver                   │
│  将 Decision 转化为目标仓位                     │
│  RankingResolver / TimingResolver              │
├──────────────────────────────────────────────┤
│              执行层 Executor                   │
│  BacktestExecutor（模拟撮合）                  │
│  LiveExecutor（推送通知）                      │
├──────────────────────────────────────────────┤
│              数据层 DataFeed                   │
│  K线 / 宏观指标 / 爬虫数据                     │
│  本地缓存（增量更新）                           │
├──────────────────────────────────────────────┤
│              通知层 Notifier                   │
│  企业微信 / 邮件                               │
├──────────────────────────────────────────────┤
│              研究层 Research                   │
│  FactorLab / Validator / 回测引擎              │
├──────────────────────────────────────────────┤
│              指标层 Indicator                  │
│  纯函数：DataFrame → 附加指标列的 DataFrame     │
└──────────────────────────────────────────────┘
```

### 2.2 核心设计原则

**原则一：策略只产出 Decision，不执行交易**

Decision 是语义丰富的决策对象，不是简单的买/卖信号。当前支持两种决策类型：

- **ROTATION**：多标的轮动——产出排序优先级和权重建议
- **TIMING**：单标的择时——产出入场/出场/观望决策

未来如需支持宏观择时、因子暴露、事件驱动等类型，只需扩展 DecisionType 枚举和 Decision 字段，不需要修改核心接口。

**原则二：数据层是多源异构的**

策略的数据需求远不止K线。DataFeed 通过 DataRequest.data_type 区分数据类型，支持日K、周K、分钟K、宏观指标、爬虫数据等。未来新数据源只需实现 DataFeed 接口。

**原则三：回测与实盘共用策略代码**

区别仅在 DataFeed（缓存 vs API）和 Executor（模拟 vs 推送）。策略代码、Decision 产出、Resolver 决议——完全相同。

**原则四：验证是策略上线的必经之路**

框架提供 FactorLab（IC 分析、分层回测）和 Validator（参数扫描、Walk-Forward），但判定标准由策略研究者定义。

### 2.3 核心接口定义

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from datetime import datetime
from enum import Enum
from typing import Any, Optional
import pandas as pd


# ============ 决策模型 ============

class DecisionType(Enum):
    ROTATION = "rotation"
    TIMING = "timing"


@dataclass
class Decision:
    """
    策略产出的决策对象。

    策略→Resolver 之间的核心契约。
    Decision 不等于交易指令——Resolver 负责将 Decision 转化为 TargetPosition。
    """
    decision_type: DecisionType
    timestamp: datetime
    reason: str

    target_code: str = ""
    direction: str = ""        # 'enter' | 'exit' | 'hold'
    weight: float = 0.0        # 目标权重（0.0~1.0），轮动策略使用
    priority: int = 0          # 排序优先级，轮动策略使用
    confidence: float = 1.0    # 置信度

    extra: dict[str, Any] = field(default_factory=dict)
    strategy_name: str = ""
    indicator_values: dict[str, float] = field(default_factory=dict)


# ============ 数据层 ============

@dataclass
class DataRequest:
    """
    数据请求描述。

    data_type 标识数据类型：
    - 'daily_k': 日K线
    - 'weekly_k': 周K线
    - 'monthly_k': 月K线
    - 'minute_k': 分钟K线
    - 'macro': 宏观指标（需在 params 中指定具体指标名）
    - 'scraper_jisilu': 集思录 QDII 溢价率
    - 'scraper_chinabond': 中债国债收益率
    - 'scraper_10jqka': 同花顺 AH 溢价
    - 'scraper_jijinhao': 基金号国债期货
    - 'custom': 自定义数据源
    """
    codes: list[str]
    data_type: str
    start: str
    end: str
    params: dict[str, Any] = field(default_factory=dict)


@dataclass
class DataResponse:
    """
    数据响应——标准化后的数据容器。

    bar_data: 标的关联数据，{code: DataFrame}
    macro_data: 非标的关联数据（宏观指标等）
    """
    bar_data: dict[str, pd.DataFrame] = field(default_factory=dict)
    macro_data: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)


class DataFeed(ABC):
    @abstractmethod
    def get_data(self, request: DataRequest) -> DataResponse:
        pass

    @abstractmethod
    def get_latest_price(self, code: str) -> float:
        pass


class CachedDataFeed(DataFeed):
    """带本地缓存的 DataFeed，增量更新"""
    def __init__(self, source: DataFeed, cache_dir: str):
        self.source = source
        self.cache_dir = cache_dir


# ============ 指标层 ============

class Indicator(ABC):
    """纯函数：输入 DataFrame，输出附加指标列的 DataFrame"""
    @abstractmethod
    def compute(self, data: pd.DataFrame, **kwargs) -> pd.DataFrame:
        pass


# ============ 策略层 ============

class Strategy(ABC):
    @abstractmethod
    def produce_decisions(self,
                          data: DataResponse,
                          positions: dict[str, Any]) -> list[Decision]:
        pass

    @abstractmethod
    def get_required_data(self) -> list[DataRequest]:
        pass

    @property
    @abstractmethod
    def name(self) -> str:
        pass

    @property
    @abstractmethod
    def config(self) -> 'StrategyConfig':
        pass


# ============ 组合层 ============

@dataclass
class TargetPosition:
    """目标仓位——Resolver 产出的可执行仓位方案"""
    code: str
    target_weight: float
    reason: str


class Resolver(ABC):
    @abstractmethod
    def resolve(self,
                decisions: list[Decision],
                current_positions: dict[str, Any],
                available_capital: float) -> list[TargetPosition]:
        pass


class RankingResolver(Resolver):
    """轮动策略专用：TOP_K 选择 + 权重分配"""
    def __init__(self, top_k: int, weight_method: str = 'equal'):
        self.top_k = top_k
        self.weight_method = weight_method  # 'equal' | 'signal_weight' | 'kelly'


class TimingResolver(Resolver):
    """择时策略专用：入场/出场 → 仓位 0 或 1"""
    pass


# ============ 执行层 ============

class Executor(ABC):
    @abstractmethod
    def execute(self, targets: list[TargetPosition], data: DataResponse) -> dict:
        pass

    @abstractmethod
    def get_positions(self) -> dict[str, Any]:
        pass


class BacktestExecutor(Executor):
    """回测执行器——模拟撮合，含交易成本建模"""
    def __init__(self, initial_capital: float,
                 commission_rate: float = 0.00025,
                 min_commission: float = 5.0,
                 slippage: float = 0.001):
        ...


class LiveExecutor(Executor):
    """实盘执行器——生成交易建议文本推送通知，持仓持久化到 JSON"""
    def __init__(self, notifier, position_file: str):
        ...


# ============ 通知层 ============

class Notifier(ABC):
    @abstractmethod
    def notify(self, title: str, content: str, level: str = "info"):
        pass


# ============ 策略配置 ============

@dataclass
class StrategyConfig:
    """策略参数容器基类。每个策略定义自己的子类。"""
    strategy_name: str = ""

    def to_dict(self) -> dict:
        return {k: v for k, v in self.__dict__.items() if not k.startswith('_')}

    @classmethod
    def from_dict(cls, d: dict) -> 'StrategyConfig':
        return cls(**{k: v for k, v in d.items() if k in cls.__dataclass_fields__})
```

### 3.4 数据刷新机制

API 有缓存，获取数据前必须先调用一次"刷新"请求，否则可能拿到旧数据：

- 日K线刷新：`startDate` = 10天前，`type` = 1
- 周K线刷新：`startDate` = 20天前，`type` = 1
- 月K线刷新：`startDate` = 60天前，`type` = 1

刷新请求不需要检查响应内容，只需确保 HTTP 200。

### 3.5 当日数据补全

如果当天是交易日但日K线数据缺失最后一天（API 延迟），需要从分时数据补全：

```python
# 判断是否只缺今天
# 从最后一条数据的日期到今天之间，如果没有其他交易日，则只缺今天
# 补全方式：从分时数据获取今天的 open, price(作为close), high, low, volume
```

### 3.6 增量数据更新

**核心原则**：新数据只补充到本地缓存，不覆盖已有数据。

**本地缓存格式**：每个基金一个 JSON 文件，文件名 `fund_data_{code}.json`，内容为 `[{"date": "2020-01-02", "open": 1.234, ...}, ...]`。

**参数追踪文件**：`batch_params.json`，记录每个基金已缓存数据的实际日期范围：

```json
{
  "report_date": "2024-01-15",
  "days": 5000,
  "fund": {"159915": "创业板ETF", ...},
  "target_start_date": "2010-01-01",
  "target_end_date": "2024-01-15",
  "fund_actual_date_ranges": {
    "159915": {"min_date": "2011-01-05", "max_date": "2024-01-15"},
    "510300": {"min_date": "2012-05-28", "max_date": "2024-01-15"}
  }
}
```

**增量更新逻辑**：

```
对每个基金 code：
  1. 如果 code 不在参数文件的 fund 字段中 → 完整更新
  2. 如果 fund_actual_date_ranges[code].min_date > target_start_date → 完整更新
  3. 如果 target_end_date - fund_actual_date_ranges[code].max_date > 7天 → 增量更新
  4. 否则 → 不需要更新

完整更新：获取 days 天数据
增量更新：获取 (target_end_date - max_date + 15天) 的数据

合并规则：
  merged = pd.concat([existing_df, new_df])
  merged = merged.drop_duplicates(subset=['date'], keep='last')
  merged = merged.sort_values('date')
```

### 3.8 数据预处理（日期对齐）

多基金回测时，需要将所有基金对齐到同一日期序列：

```python
# 以第一个基金的日期序列为模板
tmpl_dates = dataframes[0]['date'].tolist()

for idx, fund_df in enumerate(dataframes):
    aligned_df = fund_df.set_index('date').reindex(tmpl_dates)
    aligned_df = aligned_df.ffill().reset_index()

    for col in aligned_df.columns:
        if col != 'date' and aligned_df[col].isna().any():
            first_valid = aligned_df[col].first_valid_index()
            if first_valid is not None:
                aligned_df.loc[:first_valid - 1, col] = aligned_df.loc[first_valid, col]

    dataframes[idx] = aligned_df
```

**已知问题**：`ffill` 会引入偏差——停牌日也用前值填充，回测时假设可以交易但实际不能。在数据质量检查模块中需要标记停牌日（成交量为0的日期）。

## 四、策略层详细规格

### 4.1 Strategy 接口

```python
class Strategy(ABC):
    @abstractmethod
    def produce_decisions(self, data: DataResponse, positions: dict) -> list[Decision]:
        """核心接口：接收数据+当前持仓，产出决策列表"""

    @abstractmethod
    def get_required_data(self) -> list[DataRequest]:
        """声明策略需要什么数据——调度层据此提前准备"""

    @property
    @abstractmethod
    def name(self) -> str:
        """策略名称——日志追溯和多策略并行时的标识"""

    @property
    @abstractmethod
    def config(self) -> StrategyConfig:
        """策略参数容器——参数扫描和持久化"""
```

### 4.2 策略与 Resolver 的解耦

策略只产出 Decision，Resolver 将 Decision 转化为 TargetPosition。这使得：

- 同一策略可以搭配不同 Resolver（等权、Kelly权重等）
- 策略研究者专注策略逻辑，不需要操心仓位管理
- 组合优化可以独立演进，不影响策略代码

### 4.3 现有策略适配验证

**持仓管理**：

```python
# 买入
def buy(self, code, shares, price):
    actual_price = price * (1 + self.slippage)
    trade_value = shares * actual_price
    commission = max(trade_value * self.commission_rate, self.min_commission)
    total_cost = trade_value + commission

    if total_cost > self.cash:
        # 资金不足，按可用资金调整份额
        shares = int((self.cash - self.min_commission) / (actual_price * (1 + self.commission_rate)) / 100) * 100
        if shares <= 0:
            return
        trade_value = shares * actual_price
        commission = max(trade_value * self.commission_rate, self.min_commission)
        total_cost = trade_value + commission

    self.cash -= total_cost
    if code in self.positions:
        # 追加买入：加权平均成本
        old = self.positions[code]
        total_shares = old['shares'] + shares
        old_avg = old['avg_cost']
        new_avg = (old['shares'] * old_avg + shares * actual_price) / total_shares
        self.positions[code] = {'shares': total_shares, 'avg_cost': new_avg,
                                'high_watermark': max(old['high_watermark'], actual_price)}
    else:
        self.positions[code] = {'shares': shares, 'avg_cost': actual_price,
                                'high_watermark': actual_price}

# 卖出
def sell(self, code, shares, price):
    actual_price = price * (1 - self.slippage)
    trade_value = shares * actual_price
    commission = max(trade_value * self.commission_rate, self.min_commission)

    self.cash += trade_value - commission
    del self.positions[code]
```

**A股交易规则**：

- 最小交易单位：100 份（1手），买入份额必须为 100 的整数倍
- T+1：当日买入次日方可卖出（回测中暂不强制，实盘需注意）

#### 9.5.2 git_pull.py — 代码同步

```python
import os, time, git
from git import Repo
from core.notifier import WeChatNotifier

notifier = WeChatNotifier()

def pull_from_remote(repo_path, branch_name) -> tuple[bool, str]:
    """
    从远端仓库拉取指定分支最新代码。

    最多重试 10 次，每次失败等待 30 秒。
    成功后通过微信发送更新信息（含新增提交数量和提交详情）。
    """
    try:
        repo = Repo(repo_path)
        origin = repo.remotes.origin

        if branch_name not in repo.heads:
            repo.create_head(branch_name, origin.refs[branch_name])

        prev_commit = repo.head.commit
        repo.git.checkout(branch_name)
        origin.pull()

        latest_commit = repo.head.commit
        if prev_commit == latest_commit:
            return True, f"分支 {branch_name} 无新提交"

        log_output = repo.git.log(
            f"{prev_commit.hexsha}..{latest_commit.hexsha}",
            "--pretty=format:%h - %s (%an, %ad)",
            "--date=short"
        )
        commits_list = log_output.strip().splitlines()
        commit_details = "\n".join([f"  • {c}" for c in commits_list])
        return True, f"拉取 {branch_name}，新增 {len(commits_list)} 个提交\n{commit_details}"

    except git.exc.GitCommandError as e:
        if "conflict" in str(e):
            return False, f"代码冲突，需手动解决\n{e}"
        return False, f"拉取失败: {e.__class__.__name__}"

if __name__ == "__main__":
    report_text = ""
    for i in range(10):
        result, git_info = pull_from_remote(os.getcwd(), "main")
        report_text = git_info
        if result:
            break
        time.sleep(30)
    notifier.notify("代码同步", report_text)
```

#### 9.5.8 wait_until 时间等待函数

```python
import time
from datetime import datetime

def wait_until(hour: int, minute: int, second: int = 0):
    """
    等待到指定时间。

    已知问题：当目标时间已过时，计算出的 sleep_time 为负数，
    会导致 time.sleep() 抛出 ValueError。
    调用方需要 try-except 捕获此异常。

    正确行为应该是：如果目标时间已过，立即返回（不等待）。
    新工程中应修复此问题。
    """
    target = datetime.now().replace(hour=hour, minute=minute, second=second,
                                     microsecond=0)
    now = datetime.now()
    if now >= target:
        return  # 目标时间已过，立即返回
    sleep_time = (target - now).total_seconds()
    time.sleep(sleep_time)
```

### 10.2 Kelly 公式

```
f* = (bp - q) / b

其中：
  b = avg_win / avg_lose  （盈亏比）
  p = win_rate            （胜率）
  q = 1 - p               （败率）

返回值含义：建议投入总资金的比例，范围 [0, 1)
```

## 十五、MyTT 库依赖

MyTT 库是可选依赖。所需函数可以内联：

```python
def MA(S, N):
    return pd.Series(S).rolling(N).mean().values

def ROC(CLOSE, N=12, M=6):
    roc = 100 * (CLOSE - REF(CLOSE, N)) / REF(CLOSE, N)
    maroc = MA(roc, M)
    return roc, maroc

def REF(S, N=1):
    return pd.Series(S).shift(N).values

def BOLL(CLOSE, N=20, P=2):
    MID = MA(CLOSE, N)
    UPPER = MID + P * STD(CLOSE, N)
    LOWER = MID - P * STD(CLOSE, N)
    return UPPER, MID, LOWER

def ATR(HIGH, LOW, CLOSE, N=20):
    TR = MAX(MAX((HIGH - LOW), ABS(REF(CLOSE, 1) - HIGH)), ABS(REF(CLOSE, 1) - LOW))
    return MA(TR, N)
```

也可以直接 `pip install MyTT`。框架不强制。

## 十七、关键注意事项

### 17.1 必须避免的设计错误

| 错误 | 正确做法 |
|---|---|
| Strategy 输出 Signal（买/卖/持有） | Strategy 输出 Decision（语义丰富的决策对象） |
| 为每种策略特化 Strategy 接口 | Strategy 接口统一为 produce_decisions()，通过 DecisionType 区分 |
| DataFeed 只支持 K线 | DataFeed 支持任意 DataRequest，通过 data_type 区分 |
| 策略直接操作账户 | 策略只产出 Decision，Resolver→Executor 链条处理交易 |
| 参数硬编码在策略代码中 | StrategyConfig 数据类，支持参数扫描 |

### 17.2 框架扩展性检查

每当添加新策略时，检查：

1. **是否需要修改核心接口？** → 如果需要，说明接口不够通用，应扩展而非特化
2. **新策略是否能复用现有 Resolver？** → 如果不能，考虑是否需要新增 Resolver 类型
3. **新数据源是否能接入现有 DataFeed 体系？** → 如果不能，考虑扩展 DataRequest.data_type
4. **新策略是否能在回测主循环中直接回测？** → 如果不能，说明回测引擎不够通用

### 17.3 现有策略迁移策略

- 现有策略的核心计算逻辑（指标计算、决策规则）保持不变
- 只需将策略的输出从"直接操作账户"改为"产出 Decision"
- 只需将策略的数据获取从"各自调用 API"改为"通过 DataFeed 抽象获取"
- 组合管理和风控逻辑从策略内部提取为独立的 Resolver

### 17.4 API 错误处理

```
1. HTTP 状态码非 200 → 抛异常，触发重试
2. 响应 JSON 的 code 字段非 200 → 抛异常，记录 message 和 traceId
3. 连续 3 次重试失败 → 向上抛出异常
4. 当日数据缺失且无法从分时补全 → 抛异常
5. 资金不足（现金为负）→ 抛异常
```
