# QuantForge 演进策略

> **定位**：本文档描述 QuantForge 从 L3 精简版到全版愿景的演进路径。核心原则是**按需扩展**——只有当现有架构无法满足新需求时才扩展，不为"将来可能用到"提前设计。
>
> **与工程规格书的关系**：工程规格书描述"当前要实现什么"，本文档描述"何时扩展、如何扩展、扩展到什么程度"。

---

## 一、演进总览

```
L3 精简版（当前）
  │
  ├── 触发条件满足 → L3.5 增强版
  │     │
  │     ├── 触发条件满足 → L4 专业版
  │     │     │
  │     │     └── 触发条件满足 → L4+ 全版愿景
  │     │
  │     └── 也可以跳过，直接从 L3 到 L4
  │
  └── 也可以一直停留在 L3，如果需求不变
```

**关键原则**：

1. **每一级都是完整可工作的系统**，不是半成品
2. **升级由需求驱动**，不是由时间驱动
3. **升级是增量式的**，不需要重写已有代码
4. **升级前先评估**：新需求能否在当前架构中用简单方式满足？只有确实无法满足时才扩展核心接口

## 二、各级别定义

### L3 — 精简版（当前实施）

**能力范围**：

| 模块 | 能力 | 限制 |
|---|---|---|
| Decision | ROTATION + TIMING 两种类型 | 宏观状态、因子暴露、事件驱动等类型未支持 |
| Resolver | RankingResolver + TimingResolver | 无独立风控链条，止损硬编码在 Resolver 中 |
| DataFeed | SinaFinanceFeed + AutoStockFeed + WebScraperFeed + CachedDataFeed | 无 CompositeDataFeed，多源数据需手动组合 |
| Executor | BacktestExecutor + LiveExecutor | 无 SilentExecutor |
| Research | IC分析 + 分层回测 + 参数扫描 + Walk-Forward | 无收益分解、无成本敏感性分析 |
| Notifier | WeChat + Email | 无 CompositeNotifier |
| Scheduler | 回测 + 实盘监控 | 无独立研究调度器 |

**能支持的策略类型**：

- ✅ 多标的轮动（ROC动量）
- ✅ 单标的择时（红利低波、国债30）
- ✅ 宏观择时（股债利差、AH溢价——归入TIMING）
- ✅ 事件驱动（日内半仓——归入TIMING）
- ✅ 相对价值（QDII溢价——归入ROTATION+TIMING）

### L3.5 — 增强版

**在 L3 基础上增加**：

| 模块 | 新增能力 | 触发条件 |
|---|---|---|
| RiskGuard | 独立风控链条 | 止损逻辑超过3种，或需要在多个Resolver间共享风控规则 |
| CompositeDataFeed | 多源数据自动组合 | 策略需要同时从3个以上数据源获取数据 |
| FactorLab | 因子换手率 + 因子相关性 | 开始做多因子组合研究 |
| Validator | 收益分解 + 成本敏感性 | 需要区分Alpha和Beta收益来源 |
| BacktestAnalyzer | Sortino比率 + Calmar比率 | 需要更精细的风险调整收益指标 |

### L4 — 专业版

**在 L3.5 基础上增加**：

| 模块 | 新增能力 | 触发条件 |
|---|---|---|
| Decision | MACRO_STATE + FACTOR_EXPOSURE + EVENT 类型 | 策略产出无法用 ROTATION/TIMING 表示 |
| Resolver | MacroResolver + OptimizationResolver | 新 DecisionType 需要对应的 Resolver |
| StrategyRegistry | 策略注册与发现 | 同时运行的策略超过5个，需要统一管理 |
| UniverseManager | 标的池统一管理 | 多策略共享标的池，需要统一数据缓存 |
| FactorLab | 因子正交化 + 动态因子权重 | 多因子组合需要去除共线性 |
| Validator | 蒙特卡洛模拟 + Bootstrap | 需要更严格的统计显著性检验 |

### L4+ — 全版愿景

**在 L4 基础上增加**：

| 模块 | 新增能力 | 触发条件 |
|---|---|---|
| Decision | CUSTOM 类型 + 完全自定义字段 | 策略产出完全无法用预定义类型表示 |
| Resolver | 任意自定义 Resolver | 需要全新的组合优化算法 |
| DataFeed | 另类数据 + 流式数据 + 数据库源 | 接入非传统数据源 |
| Executor | 真实下单执行器 | 需要自动化交易（非仅推送建议） |
| Research | ML辅助因子挖掘 + 在线学习 | 机器学习成为策略研发的核心手段 |
| RiskGuard | 尾部风险 + 压力测试 + VaR/CVaR | 需要机构级风险管理 |
| Scheduler | 事件驱动调度 + 异步调度 | 需要实时响应市场事件 |

## 三、各阶段详细演进路径

### Phase 1：核心骨架 + ROC 策略落地（2-3 周）

**目标**：验证框架核心接口是否足够通用。

**实施步骤**：

1. 搭建目录结构
2. 实现 Decision + DecisionType（只定义 ROTATION 和 TIMING）
3. 实现 DataRequest + DataResponse + DataFeed + CachedDataFeed
4. 实现 AutoStockFeed（K线数据源，含刷新、当日补全、重试）
5. 实现 Indicator 基类 + 技术指标（ROC, MA, REF）
6. 实现 Strategy 基类
7. 实现 ROCConfig + ROCStrategy.produce_decisions()
8. 实现 Resolver 基类 + RankingResolver（含止损逻辑）
9. 实现 BacktestExecutor（含交易成本建模）
10. 实现回测主循环 + BacktestAnalyzer
11. 实现 main_backtest.py
12. 运行 ROC 策略回测，对比旧系统结果

**完成标志**：ROC 策略在新框架中回测结果与旧系统一致。

**关键验证**：
- ROC 策略的 Decision 产出是否顺畅？是否需要修改 Decision 接口？
- RankingResolver 是否能完整处理 TOP_K + 权重分配 + 止损？
- BacktestExecutor 的交易成本建模是否准确？

### Phase 2：验证基础设施（1-2 周）

**目标**：策略上线前能通过统计验证。

**实施步骤**：

13. 实现 FactorLab（IC 分析 + 分层回测）
14. 实现 Validator（参数扫描 + Walk-Forward）
15. 用 ROC 策略跑完整验证管道
16. 根据验证结果调整策略参数

**完成标志**：ROC 策略通过完整验证管道（IC有效 + 参数稳健 + Walk-Forward正收益 + 扣成本后盈利）。

**关键验证**：
- IC 分析结果是否与直觉一致？
- 参数扫描是否发现稳健参数区域（非孤岛）？
- Walk-Forward 的 OOS 收益是否为正？

### Phase 3：更多策略 + 实盘监控（1-2 周）

**目标**：验证框架对不同策略类型的通用性，完成实盘闭环。

**实施步骤**：

17. 实现 TimingResolver
18. 实现红利低波或30年国债择时策略 → 验证 TIMING 类型
19. 实现 WebScraperFeed（爬虫数据源）
20. 实现股债利差宏观策略 → 验证多源数据接入
21. 实现 WeChatNotifier + EmailNotifier
22. 实现 LiveExecutor（持仓持久化 + 推送通知）
23. 实现 main_monitor.py + 多策略并行监控

**完成标志**：至少 3 种不同类型策略在新框架中运行，实盘监控可用。

**关键验证**：
- TimingResolver 是否能处理单标的择时的入场/出场？
- WebScraperFeed 是否能统一接入不同爬虫数据源？
- 宏观策略（股债利差）产出 TIMING 类型 Decision 是否自然？
- 实盘监控是否稳定运行？

### Phase 4：系统完善（1-2 周）

**目标**：补充边界场景，提升系统健壮性。

**实施步骤**：

24. 实现日内半仓策略 → 验证分钟K线数据接入
25. 实现回测可视化（matplotlib）
26. 实现基金摩擦分析
27. 性能优化（数据缓存、并行计算）
28. 边界场景测试（停牌日、资金不足、API异常等）

**完成标志**：L3 精简版功能完整，所有现有策略都能在新框架中运行。

## 四、L3 → L3.5 升级指南

### 4.1 触发条件

**以下任一条件满足时，考虑升级到 L3.5**：

1. **止损逻辑超过3种**：当前止损硬编码在 Resolver 中，如果新增止损规则导致 Resolver 代码膨胀
2. **策略需要3个以上数据源**：当前多源数据需手动组合，如果组合逻辑复杂
3. **开始做多因子组合研究**：需要因子换手率和因子相关性分析
4. **需要区分Alpha和Beta**：需要收益分解和成本敏感性分析

### 4.2 升级步骤

**Step 1：提取 RiskGuard（如果触发条件1满足）**

```python
# 从 Resolver 中提取止损逻辑
class RiskGuard(ABC):
    @abstractmethod
    def guard(self, targets, positions, data) -> list[TargetPosition]:
        pass

class HighWatermarkStopLoss(RiskGuard): ...
class CostStopLoss(RiskGuard): ...
class MaxPositionGuard(RiskGuard): ...
class DrawdownStopLoss(RiskGuard): ...

# Resolver 不再包含止损逻辑，改为调用 RiskGuard 链条
class RankingResolver(Resolver):
    def __init__(self, top_k, weight_method, risk_guards: list[RiskGuard]):
        self.risk_guards = risk_guards

    def resolve(self, decisions, positions, capital):
        targets = self._compute_targets(decisions, positions, capital)
        for guard in self.risk_guards:
            targets = guard.guard(targets, positions, data)
        return targets
```

**影响范围**：Resolver 接口不变（只是内部实现变了），策略代码不变，Executor 不变。

**Step 2：实现 CompositeDataFeed（如果触发条件2满足）**

```python
class CompositeDataFeed(DataFeed):
    def __init__(self, feeds: dict[str, DataFeed]):
        self.feeds = feeds  # {data_type_prefix: DataFeed}

    def get_data(self, request: DataRequest) -> DataResponse:
        feed = self._select_feed(request.data_type)
        return feed.get_data(request)
```

**影响范围**：DataFeed 接口不变，只是新增一个组合实现。策略代码不变。

**Step 3：扩展 FactorLab（如果触发条件3满足）**

```python
class FactorLab:
    # 已有
    def compute_ic(self, ...): ...
    def layered_backtest(self, ...): ...

    # 新增
    def factor_turnover(self, factor_values, periods=5):
        """因子换手率——衡量因子稳定性"""

    def factor_correlation(self, factor_a, factor_b):
        """因子间相关性——多因子组合时避免冗余"""
```

**影响范围**：纯新增方法，不影响已有代码。

**Step 4：扩展 Validator（如果触发条件4满足）**

```python
class Validator:
    # 已有
    def parameter_sweep(self, ...): ...
    def walk_forward(self, ...): ...

    # 新增
    def alpha_decomposition(self, strategy_returns, benchmark_returns):
        """收益分解：Alpha = 策略收益 - Beta × 基准收益"""

    def cost_sensitivity(self, strategy_class, config, cost_range, data):
        """交易成本敏感性分析"""
```

**影响范围**：纯新增方法，不影响已有代码。

## 五、L3.5 → L4 升级指南

### 5.1 触发条件

**以下任一条件满足时，考虑升级到 L4**：

1. **策略产出无法用 ROTATION/TIMING 表示**：例如纯宏观状态判断（不关联任何标的）、因子暴露向量（需要组合优化）、自定义事件信号
2. **同时运行的策略超过5个**：需要统一注册和管理
3. **多策略共享标的池**：需要统一数据缓存管理
4. **多因子组合需要正交化**：因子间共线性严重

### 5.2 升级步骤

**Step 1：扩展 DecisionType（如果触发条件1满足）**

```python
class DecisionType(Enum):
    ROTATION = "rotation"
    TIMING = "timing"
    # 新增
    MACRO_STATE = "macro_state"        # 纯宏观状态，不关联标的
    FACTOR_EXPOSURE = "factor_exposure"  # 因子暴露向量
    EVENT = "event"                      # 事件驱动
```

**关键**：Decision 数据类不需要修改——新类型通过已有的 `extra` 字段和可选字段承载信息。只有当现有字段确实无法表达新类型的语义时，才添加新字段。

**Step 2：新增 Resolver（如果触发条件1满足）**

```python
class MacroResolver(Resolver):
    """宏观策略：市场状态 → 全仓/半仓/空仓"""
    def __init__(self, state_position_map: dict[str, float]):
        self.state_position_map = state_position_map

class OptimizationResolver(Resolver):
    """因子策略：因子暴露 → 组合优化"""
    pass
```

**Step 3：实现 StrategyRegistry（如果触发条件2满足）**

```python
class StrategyRegistry:
    _strategies: dict[str, Strategy] = {}

    def register(self, strategy: Strategy): ...
    def get(self, name: str) -> Strategy: ...
    def all(self) -> list[Strategy]: ...
```

**Step 4：实现 UniverseManager（如果触发条件3满足）**

```python
class UniverseManager:
    def register_universe(self, strategy_name, codes): ...
    def get_all_codes(self) -> list[str]: ...
    def ensure_data_available(self, requests): ...
```

**Step 5：扩展 FactorLab（如果触发条件4满足）**

```python
class FactorLab:
    # 新增
    def factor_orthogonalize(self, factors, method='gram_schmidt'):
        """因子正交化——去除共线性"""

    def dynamic_factor_weight(self, factors, returns, lookback=60):
        """动态因子权重——根据因子近期表现调整"""
```

## 七、升级决策框架

每当遇到新需求时，按以下流程决策：

```
新需求出现
  │
  ├── 能否在当前架构中用简单方式满足？
  │     │
  │     ├── 能 → 直接实现，不升级架构
  │     │
  │     └── 不能 → 是否满足某个升级触发条件？
  │           │
  │           ├── 满足 → 执行对应的升级步骤
  │           │
  │           └── 不满足 → 重新评估：
  │                 │
  │                 ├── 需求是否真的必要？→ 可能不需要实现
  │                 │
  │                 └── 确实必要 → 扩展核心接口，更新工程规格书
  │
  └── 评估标准：
        1. 新增代码是否超过当前模块的30%？
        2. 是否需要修改核心接口（Decision, Strategy, Resolver, DataFeed）？
        3. 是否引入了新的概念（如RiskGuard、Registry等）？
        4. 是否改变了已有模块的行为？
```

## 八、架构扩展模式

以下是各模块的标准扩展模式，升级时按此模式操作：

### 8.1 Decision 扩展

```
1. 在 DecisionType 枚举中添加新类型
2. 评估 Decision 数据类的字段是否足够：
   - 足够 → 只加枚举值
   - 不足够 → 添加新的 Optional 字段（不破坏已有代码）
   - 严重不足 → 使用 extra 字段或创建 Decision 子类
3. 实现对应的 Resolver
4. 更新回测主循环（如果需要）
```

### 8.2 DataFeed 扩展

```
1. 在 DataRequest.data_type 中添加新类型标识
2. 实现 DataFeed 子类
3. 在 CachedDataFeed 中添加对应的缓存逻辑（如果需要）
4. 策略通过 get_required_data() 声明需要新数据类型
```

### 8.3 Resolver 扩展

```
1. 实现 Resolver 子类
2. 在策略配置中指定使用哪个 Resolver
3. 不修改已有 Resolver 的接口
```

### 8.4 Research 扩展

```
1. 在 FactorLab 或 Validator 中添加新方法
2. 纯新增，不修改已有方法
3. 在验证管道中添加新步骤（如果需要）
```

## 十、风险与应对

### 10.1 过度设计风险

**表现**：在 L3 阶段就实现了 L4 的功能。

**应对**：严格遵守触发条件。每次添加新功能前问自己："当前是否有具体需求驱动这个功能？"

### 10.2 接口不稳定风险

**表现**：频繁修改核心接口（Decision, Strategy, Resolver, DataFeed）。

**应对**：
- 核心接口一旦定义，尽量不修改
- 通过 Optional 字段和 extra 字段扩展，而非修改已有字段
- 如果确实需要修改，先评估影响范围，确保向后兼容

### 10.3 策略迁移风险

**表现**：从旧系统迁移策略时，结果不一致。

**应对**：
- Phase 1 的完成标志就是"ROC 策略回测结果与旧系统一致"
- 迁移时保持核心计算逻辑不变，只改接口适配
- 对比新旧系统的中间结果（指标值、信号、交易记录），逐步排查差异

### 10.4 实盘稳定性风险

**表现**：实盘监控进程崩溃或推送错误信息。

**应对**：
- 实盘前在回测模式充分验证
- 实盘初期保留旧系统并行运行，对比结果
- 完善异常处理和日志记录
- 关键操作（推送通知、持仓持久化）加 try-catch

## 十一、与专业量化之路的对应关系

| 级别 | 对应专业量化之路 | 核心能力 | 研究重点 |
|---|---|---|---|
| L3 | Level 2+ → Level 3 | 统计严谨化：交易成本 + 因子检验 + Walk-Forward | 验证现有策略有效性 |
| L3.5 | Level 3 → Level 3.5 | 因子体系：多因子组合 + Alpha/Beta 分解 | 从单因子到多因子 |
| L4 | Level 3.5 → Level 4 | 组合优化：多策略组合 + 风险模型 + 动态仓位 | 多策略协同 |
| L4+ | Level 4+ | ML辅助 + 机构风控 + 自动化交易 | 机器学习增强 |

每个级别对应的知识体系：

- **L3**：《因子投资》第3-5章（因子检验）+ 交易成本建模
- **L3.5**：《主动投资组合管理》第6-8章（信息率）+ 多因子组合
- **L4**：《主动投资组合管理》完整 + Barra风险模型
- **L4+**：《Advances in Financial ML》+ 衍生品定价（如需要）
