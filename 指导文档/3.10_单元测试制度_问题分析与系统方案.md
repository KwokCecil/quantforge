# QuantForge 单元测试制度 — 问题分析与系统方案

> **创建日期**：2026-05-13
> **目的**：分析项目中反复出现的 bug 类型，建立分层验证制度，让错误从"事后涌现"变成"编码当即发现"。
> **适用范围**：所有 `core/` 和 `strategies/` 下模块的修改与新增。

---

## 一、背景

项目中反复出现一种模式：**Spec → 写代码 → 跑回测 → 看收益曲线 → 发现不对劲 → 反向排查**。这个流程没有中间验证环节，无法确定"已经发现了全部问题"。事实也确实是一直有新问题涌现。

以最近一次 Executor 买入逻辑修复为例：
- BacktestExecutor 和 LiveExecutor 的买入算法产生了分歧（目标市值 = 新增资金 + 已有市值 vs 仅考虑新增资金）
- 循环内递减 `available_cash` 导致后面标的分配缩水
- 两个 bug 都通过了语法检查、不报异常、能在日志中正常输出——它们只是**语义上错了**，用回测净值曲线几乎发现不了

这类问题不是偶发的。以下将项目中遇到的 bug 归纳为四个类型，并给出对应的防御手段。

---

## 二、Bug 类型学

### 类型 A：双实现分叉（Dual-Implementation Drift）

**特征**：两个模块实现了同一套概念逻辑（如 BacktestExecutor 和 LiveExecutor 的买入），但代码独立演进，逐渐产生分歧。

**典型案例**：LiveExecutor 的买入逻辑在某个版本后变成了"目标市值仅考虑新增资金"，而 BacktestExecutor 保持着"目标市值 = 新增资金 + 已有市值"的正确做法。没有任何机制发现两个实现已经不同。

**根因**：回测和实盘是两个入口，改了回测逻辑后不一定同步改实盘逻辑。回测跑出好结果 → 认为"没问题" → 实盘上线 → 出 bug。

**防御**：**契约测试**。同一套输入 → 两个执行器 → 必须输出完全相同的结果。

### 类型 B：逻辑表面正确但语义错误（Semantic Drift）

**特征**：代码语法正确、不抛异常、跑完有输出。但计算结果在绝大多数场景下是错误的，只有极端巧合时才对。

**典型案例**：
```python
# 旧代码（语义错误）
target_value = available_cash * weight_ratio    # 5,916 元（只是新增额度）
needed_value = target_value - current_value     # 5,916 - 7,325 = -1,409
if needed_value <= 0: continue                  # 跳过！不买！
```
这个代码能跑通、不报错、甚至逻辑看起来"有道理"（资金不够覆盖现有仓位就不买）。但**语义完全错了**——目标是让仓位达到总资产的某个比例，而不是判断新增资金能不能覆盖现有仓位。

第二个同样隐蔽的 bug：循环内递减 `available_cash` 导致后面的标的分配缩水。

**根因**：代码写出来时只看"逻辑自洽"，没有用具体数值走过一遍。等端到端跑完发现"为什么只花了 4.5% 的现金？"时才回头查。

**防御**：**单元测试 + 具体数字断言**。不测"回测收益"，测具体数值——"给定 X 持仓 + Y 资金 + Z 目标权重，执行后 cash 应该是 563.24 元，股数应该是 [8400, 8900, 1900]"。

### 类型 C：状态泄漏（State Leakage）

**特征**：测试或逻辑依赖了外部状态（真实文件、网络、缓存），导致结果不可复现，在不同环境/时间点表现不同。

**典型案例**：LiveExecutor 在 `__init__` 时调用 `_load_positions()` 读取 `position/position.json`。测试没做隔离，拿到的 `free_capital` 是 24,116（真实持仓）而不是测试期望的 40,000。

**根因**：测试环境和真实环境共用文件系统。第一个测试场景恰好符合"空仓"假设能通过，但换个时间、换个真实持仓后就挂了。

**防御**：**所有单元测试使用纯合成数据**（`_make_bar_data` 模式），不读任何硬盘文件。

### 类型 D：沉默失败（Silent Failure）

**特征**：代码没有断言，没有检查，输出什么就是什么。人需要看"回测收益差了多少"这种高维指标才能意识到有问题。

**典型案例**：LiveExecutor 代码执行完毕，log 输出正常，消息推送也发了，只买了 700 股通信 ETF。但这只是一个**偏离**，不是崩溃——净值曲线几乎看不出来。

**根因**：回测/实盘都是"跑完看净值曲线"的验证方式，发现的是**累积效应**而非**单步错误**。单步错了 5%，净值可能只差 0.5%。

**防御**：**模块级断言**。executor 执行前后对比 cash、shares，断言具体值。而不是等净值曲线出来再"感觉不对"。

## 三、方法论：从"最终现象验证"到"分层验证"

### 当前验证流程

```
Spec → 写代码 → 跑回测（数分钟） → 看收益曲线 → 异常？ → 反向溯源（数十分钟）
         ↑                                                    ↓
         └────────── 不知道有没有遗漏其他 bug ←───────── 修完一个
```

问题不在"回测应该更快"，而在于**回测验证的是端到端行为，而 bug 发生在模块内部**。用端到端方式发现模块级 bug，就像用望远镜找衣服上的线头。

### 目标验证流程

```
Spec → 写模块代码 → 跑模块单测（<1s） → 通过？ → 跑契约测试（5s） → 通过？ → 跑回测验证
          ↑               ↓ 不通过                      ↓ 不通过
          └────── 当场修复 ←────────────────── 当场修复
```

关键变化：**在回测之前加了两层快速检查**——模块单测和契约测试。每层只需要几秒，覆盖了上述 ABCD 四类 bug。

### 测试分层定义

| 层 | 测试什么 | 代表性测试 | 运行时间 | 依赖 |
|----|---------|-----------|---------|------|
| **单元测试** | 单个函数的输入→输出 | `test_new_codes_buy()`：给定 targets + 数据，产出具体 cash/shares | < 1s | 合成数据，无文件/网络 |
| **契约测试** | 两个实现的行为一致性 | `_test_executor_contract.py`：BE 和 LE 所有场景对齐 | < 5s | 合成数据，无文件/网络 |
| **集成测试** | 策略+执行器+数据的完整链路 | 现有的 `_test_roc_enhanced.py` 等 | 数十秒 | 可能需要网络/缓存 |
| **端到端** | 全周期回测 + 指标对比 | `_test_momentum_fullcycle.py` | 数分钟 | 网络/缓存 |

### 什么场景值得写单测

判断标准：**你害怕改它的那段代码**，就是该写测试的地方。

具体来说：
- 任何被**两个以上入口调用**的模块（executor、resolver、indicator）
- 包含**算术/资金计算**的逻辑（不是纯数据搬运）
- **过去出过 bug** 的地方（回归测试）
- **逻辑看起来"好像对"但用数字一走就错**的地方（语义易错点）

## 四、实施路线（渐进式）

### 阶段 1：先补"最容易再崩的"（已完成）

- [x] `_test_executor_contract.py`：BacktestExecutor vs LiveExecutor 全场景对齐（6 项场景，含空仓买入、追买、整手跳过、全部卖出、复合混合操作）
- [x] Executor 买入逻辑修复（对齐 BacktestExecutor 语义）

### 阶段 2：核心模块单元测试

用合成数据覆盖最容易出错的模块：

| 优先级 | 模块 | 测试重点 |
|--------|------|---------|
| P0 | executor | 买入/卖出/追加/整手/资金不足降级 |
| P0 | resolver | 止损/挤压卖出/权重分配边界 |
| P1 | indicator | NaN 处理/除零/空数据 |
| P1 | decision | 类型校验/数值范围 |

（具体执行可拆分到 `T010_自动化CI测试骨架` 或独立施工任务。）

### 阶段 3：建立习惯

- 每次 **改 executor 或 resolver** → 先跑对应的单测 + 契约测试
- 每次 **发现一个 bug** → 先写复现测试，再修代码
- 每次 **新增一个策略参数** → 加一个边界值测试

## 五、测试写法约定

### 文件命名

```
tests/_test_{模块名}_{类型}.py

示例：
  _test_executor_contract.py    ← 契约测试
  _test_config_system.py        ← 单元测试
  _test_roc_enhanced.py         ← 集成测试
```

### 写法规范

```python
"""一句话说明测试目的"""
import os, sys
sys.path.insert(0, ...)

# 用合成数据（DataFrame 手写），不依赖文件/网络
def _make_bar_data(codes, prices):
    ...

# 断言具体数值，不测模糊指标
def test_xxx():
    ...
    assert be.cash == 563.24, f"cash应该=563.24, 实际={be.cash}"
    assert le_pos["X"]["shares"] == 8400

# 每个测试函数独立，不依赖执行顺序
if __name__ == "__main__":
    test_xxx()
    test_yyy()
```

### 提交前检查

修改 `core/` 或 `strategies/` 下文件后：
- [ ] 运行 `tests/_test_executor_contract.py`
- [ ] 如果有覆盖到被修改模块的单测，运行它
- [ ] 确认通过后再提交

> 当前不需要全量回归测试套件，但契约测试和与被修改模块相关的单测必须跑。

## 七、备注

- 本文档是**制度设计**而非施工任务。实施阶段（阶段 2）的具体执行应拆分到独立施工任务。
