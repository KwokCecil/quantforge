# 3.12 独立第三方项目评估报告

> 评估日期：2026-05-31
> 评估范围：QuantForge 项目全部代码、测试、文档、规则体系
> 评估方式：独立第三方视角，未参考历史自评文档，完全基于代码与文档审查

---

## 一、做得很好，应继续保持发扬

### 1. Strategy → Resolver → Executor 三层解耦架构 ★★★★★

这是整个项目最闪亮的设计决策。`Strategy` 只产出语义化的 `Decision` 列表，不执行任何交易；`Resolver` 负责将决策转化为 `TargetPosition`；`Executor` 执行撮合。三者通过抽象基类契约连接，同一策略可搭配不同 Resolver（等权/Kelly/波动率倒数），同一 Resolver 可服务不同策略。

更难得的是，`DecisionType` 枚举区分了 `ROTATION`（多标的轮动）和 `TIMING`（单标的择时）两种信号类型，这使得 `MacroOverlayResolver` 可以同时消费宏观择时信号和动量排名信号，产出融合后的仓位方案——这个双层决策系统设计上非常优雅。

### 2. 条件开关系统 + 证伪标注文化 ★★★★★

`ROCConfig` 中的每个条件开关都标注了回测验证结果：

```python
STRICT_BUY: bool = False  # 🔴 严格买入：开启后Sharpe从1.10暴跌至0.26！
ROC_MA_DIRECTION: bool = False  # 🔴 开启后策略完全不产生交易信号！
```

这不是简单的注释，而是**用回测数据证伪假设**的工程文化。更值得称赞的是 `strategies/_falsified/` 目录——`voting.py`、`multifactor.py`、`rsi_factor.py` 等已被回测证伪的策略代码完整保留，每个文件都标注了"证伪依据"和"证伪原因"。这防止了未来重复踩坑，是科学研究精神的体现。

### 3. CachedDataFeed 的增量更新与前复权修正 ★★★★

`_compute_fq_correction` 将重叠区分三段独立计算比率来核验是否是纯前复权偏移，这是实盘数据管线的关键质量保证。采用了防御性的"核验不通过则仅追加新日期不全量覆盖"策略——非常稳健。

### 4. BacktestExecutor vs LiveExecutor 契约测试 ★★★★

确保回测和实盘执行器产生一致的交易结果，覆盖了空仓买入、已有持仓追买、整手不变跳过、全部卖出、dry_run 不写文件、复合场景等 6 个场景。这比单纯的单元测试更有价值。

### 5. 测试自发现机制 ★★★

通过文件首行的 `# @layer: unit|contract|integration|e2e` 标记自动分类，无需维护硬编码测试列表。新增测试文件只需遵循命名约定和首行标记即可零配置纳入。

### 6. 可视化体系 ★★★★

`BacktestComparator` 提供了标准四面板图（净值/回撤/滚动Sharpe/月度热力图）、多策略对比、A/B改进对比、参数热力图、Walk-Forward 瀑布图——这些都是量化研究中真正有用的可视化，不是花架子。

## 二、必须重视、认真整改的问题

### 1. ROCConfig 参数黑洞 —— 优先级：高

`ROCConfig` 目前有近 **50 个参数**（包括布尔开关和浮点阈值），分布在 9 个注释分组的板块中。具体问题：

**a. 缺少参数间依赖验证**

`rsi_enhance_enabled=True` 依赖于 RSI 指标被计算，但没有任何机制保证这个依赖被满足。如果用户在 JSON 预设中写错了 `rsi_period` 或者 RSI 计算失败，策略会静默地以"RSI过滤未生效"的状态运行而没有任何告警。

**b. JSON预设与代码默认值不一致的风险**

`ROCConfig` 的 `__post_init__` 有默认值（如 `inverse_vol_weight=False`），但 `tech_growth.json` 中设为 `true`。配置来源有两层（dataclass 默认值 + JSON 覆盖），理解"当前实际生效的值是什么"需要同时看两个地方。

**c. "僵尸参数"**

`sell_roc_edge=3.0` 被标记为 ★ 冗余参数，`BUY_AVERAGE` 和 `REBALANCE` 被标记为 ⚪ 无效果，但它们仍然存在于每个配置文件中。这些参数不会影响结果，但会增加认知负担。

**建议**：
- 引入配置校验层（`ROCConfig.validate()`），在策略初始化时检查参数一致性
- 将已证伪无效果的参数标记为 `deprecated` 并从预设 JSON 中移除
- 考虑将条件开关分组为独立的 `BuyFilterConfig`、`SellFilterConfig`、`RiskConfig` 子配置类

### 2. 止损逻辑三处复制 —— 优先级：高

`_check_stop_loss` 相同逻辑出现在三个类中：

| 位置 | 行数 |
|------|------|
| `RankingResolver._check_stop_loss` | 33行 |
| `MacroOverlayResolver._check_stop_loss` | 31行 |
| `TimingResolver._check_stop_loss` | 32行 |

三个实现**完全一致**。如果要修改止损逻辑（比如新增"时间止损"），需要改三个地方——遗漏任何一个都会导致行为不一致。这违反了 DRY 原则，且是真实会产生 bug 的技术债务。

**建议**：将止损逻辑提取为独立的 `StopLossChecker` 类或 `_StopLossMixin`，三个 Resolver 通过组合或继承复用。

### 3. 回测循环逻辑重复 —— 优先级：高

`backtest_core.py` 中的 `run_backtest()` 和 `main_backtest.py` 中的 `_run_rotation_backtest()` 两处实现了高度相似的回测主循环（数据加载→对齐→逐日遍历→execute）。区别只是后者多了切换策略/Resolver 的逻辑。

但项目中已有的规则明确规定了"所有回测必须走 `run_core_backtest(config)`"、"禁止重复实现回测逻辑"。而 `_run_rotation_backtest` 自己就没有遵守这个规则。

**建议**：将轮动回测的调度逻辑抽象为 `RotationScheduler` 注入到 `run_backtest` 中，而不是复制整个循环。

### 4. 缺少端到端回归测试 —— 优先级：中高

单元测试质量不错（Resolver 的四种权重模式、止损触发、top_k_sell 逻辑都有覆盖），契约测试也覆盖了 Executor 一致性。但缺少一个关键测试：**给定固定数据快照，运行完整回测流水线，断言最终净值/Sharpe 与基线一致**。

这个测试的价值在于：任何对 `backtest_core.py`、`executor.py`、`resolver.py` 的改动，如果导致回测结果变化，会在 CI 中立即暴露。目前没有这个测试，只能依赖人工跑回测来发现回归。

### 5. Monitor 模块的代码重复 —— 优先级：中

`monitors/roc_momentum_monitor.py` 中有独立的数据源初始化、缓存刷新、策略创建等逻辑，与 `main_backtest.py` 的 `run_core_backtest` 存在显著重复。虽然 Monitor 和 Backtest 是不同的使用场景，但数据管线（DataFeed 初始化 + 缓存管理 + 策略创建）可以也应该共享。

### 6. Research 目录缺少内部组织 —— 优先级：中

`research/` 目录有 35+ 个文件，混杂了：
- 真正的分析工具（`factor_lab.py`、`param_optimizer.py`、`validator.py`）
- 一次性实验脚本（`_test_momentum_fullcycle.py`、`_verify_ah_premium.py`）
- 诊断工具（`diagnose_participation.py`、`health_check.py`）

文件全部平铺，没有子目录分类。随着研究产出继续增长，这个目录会变得越来越难以导航。

### 7. 缺少类型检查 —— 优先级：中

代码中大量使用了 `Any` 类型，`positions` 的结构实际上是有明确契约的（`{'shares': int, 'avg_cost': float, 'high_watermark': float}`），但用 `Any` 抹掉了类型安全。对于量化交易系统——类型错误可能意味着错误的仓位计算——缺少 `mypy`/`pyright` 静态检查是一个隐性风险。`requirements.txt` 中没有包含类型检查工具。

### 8. 无 CI/CD —— 优先级：中

项目没有 GitHub Actions 或其他 CI 配置。虽然有 `run_all_tests.py`，但没有自动化运行。对于一个已经发展到 Phase 4 的项目，CI 的缺失是工程成熟度的一个明显缺口。

## 三、值得思考但非紧急的方向性建议

### 1. 策略参数优化的过拟合风险

`ROCConfig` 中大量的开关系统本质上是一种"离散参数网格搜索"——每个开关组合是一个独立参数点。50+ 个布尔开关意味着 2^50 的搜索空间，虽然在实践中只测试了有限组合，但"通过大量实验筛选出的最优开关组合"仍然存在过拟合到历史数据的风险。Walk-Forward 分析是很好的防御，但需要确保 OOS 窗口足够长（目前回测区间 2018-2026，约 8 年，建议 OOS 至少 2-3 年）。

### 2. 多策略并行能力的利用不足

架构设计上支持多策略并行（`strategy_name` 字段、`DecisionType` 枚举），但目前只有两个策略（ROC动量和短期反转），它们之间没有组合使用。`MacroOverlayResolver` 的概念很有潜力——同时消费 TIMING 和 ROTATION 信号——但目前在回测中没有大规模使用。

### 3. 实盘与回测的差异追踪

`LiveExecutor` 记录了 `prev_close` 用于计算当日盈亏，但没有看到"实盘执行与回测预期差异"的追踪机制。回测假设了滑点模型（固定 0.1%），实盘中真实滑点可能不同；回测假设了"按收盘价成交"，实盘中可能是盘中价格。这些差异如果长期不追踪，会侵蚀对回测结果的信任。

## 四、总结评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ★★★★★ | Strategy/Resolver/Executor 三层解耦 + DecisionType 分类，专业水准 |
| 代码质量 | ★★★★ | 核心模块质量高，但有止损复制、回测循环重复等技术债 |
| 测试覆盖 | ★★★★ | 单元+契约测试扎实，缺端到端回归测试 |
| 工程规范 | ★★★★ | 规则/技能体系完善，"证伪墓地"文化优秀 |
| 文档体系 | ★★★★ | 指导文档详尽，但 v1-v4 历史版本堆积增加维护成本 |
| CI/DevOps | ★★ | 缺少 CI/CD、类型检查、自动化门控 |
| 研究纪律 | ★★★★★ | 条件开关标注有效性、Walk-Forward 验证、参数稳健性分析，量化研究的正确范式 |

**一句话总评**：这是一个架构设计出色、研究纪律严谨、核心代码质量高的量化回测系统。当前最需要投入的不是新功能，而是**消除技术债务**（止损复制、回测循环重复）和**补全工程基础设施**（CI/CD、类型检查、端到端回归测试）。在进入更多策略/更复杂模型的阶段之前，把地基夯实比盖新楼更重要。
