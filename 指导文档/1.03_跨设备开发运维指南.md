# 1.03_跨设备开发运维指南

> **定位**：本文档指导如何使用 Git + GitHub 在开发机和运行机之间同步 QuantForge 工程代码，确保异地开发-运行的顺畅协作。
>
> **场景**：开发机（Windows，使用 Trae IDE + AI 辅助编码）↔ 远程仓库 ↔ 运行机（Windows，执行实盘监控和回测）。两台设备物理分离，通过 Git 远程仓库作为中转同步代码。

---

## 一、仓库配置

### 1.1 创建仓库

在 Git 托管平台（GitHub/Gitee 等）创建仓库：

| 项目 | 值 |
|---|---|
| 仓库名称 | `quantforge` |
| 初始化 | 不勾选 README/.gitignore（本地已有） |

### 1.2 远程仓库地址

```
https://github.com/<your-username>/quantforge.git
```

### 1.3 认证方式

推荐使用 Personal Access Token 认证（避免每次输入密码）：

1. GitHub: Settings → Developer settings → Personal access tokens → Generate new token (classic)
2. 勾选 `repo` 权限
3. 记录生成的 token

配置凭证存储（两台设备都要执行）：

```bash
git config --global credential.helper store
```

首次 push/pull 时使用 token 作为密码，之后自动缓存。

## 二、分支策略

### 2.1 分支模型

```
main          ← 稳定版本，运行机从这个分支拉取代码
  │
  ├── dev     ← 日常开发分支，AI 辅助编码在此进行
  │     │
  │     ├── feature/xxx   ← 功能分支（可选，复杂功能时使用）
  │     └── fix/xxx       ← 修复分支（可选）
  │
  └── hotfix  ← 紧急修复，直接从 main 拉出，修复合并回 main
```

### 2.2 分支规则

| 分支 | 用途 | 谁使用 | 何时合并到 main |
|---|---|---|---|
| `main` | 稳定可运行版本 | 运行机拉取 | — |
| `dev` | 日常开发 | 开发机 | Phase 完成且验证通过后 |
| `feature/*` | 功能开发 | 开发机 | 功能完成且测试通过后合并到 dev |
| `hotfix` | 紧急修复 | 开发机 | 修复后直接合并到 main + dev |

### 2.3 为什么不用单分支

旧项目只有 `master` 一个分支，开发机和运行机共用，导致：
- 开发中的半成品代码可能被运行机拉取，导致运行失败
- 运行机需要稳定版本，开发机需要频繁提交

新工程用 `main` + `dev` 分离：运行机只拉 `main`，开发机在 `dev` 上工作，Phase 完成后才合并到 `main`。

## 三、开发机工作流

### 3.1 初始化新工程

```bash
mkdir quantforge
cd quantforge
git init
git remote add origin https://github.com/<your-username>/quantforge.git

# 创建 .gitignore（见第四节）
# 创建项目文件...

git add .
git commit -m "init: QuantForge 项目初始化"
git branch -M main
git push -u origin main

# 创建 dev 分支
git checkout -b dev
git push -u origin dev
```

### 3.2 日常开发流程

```bash
# 确保在 dev 分支
git checkout dev
git pull origin dev

# ... AI 辅助编码 ...

# 提交（频繁提交，每完成一个小任务就提交）
git add <精确文件列表>
git commit -m "feat: 实现XXX功能"

# 推送到远端
git push origin dev
```

### 3.3 Phase 完成后合并到 main

```bash
# 在 dev 分支上，确保所有代码已提交
git checkout dev
git add <精确文件列表>
git commit -m "feat: Phase X 完成"

# 合并到 main
git checkout main
git merge dev
git push origin main

# 切回 dev 继续开发
git checkout dev
```

### 3.4 紧急修复

```bash
# 从 main 拉出 hotfix
git checkout main
git checkout -b hotfix/fix-xxx

# 修复代码...
git add <精确文件列表>
git commit -m "hotfix: 修复XXX问题"

# 合并回 main
git checkout main
git merge hotfix/fix-xxx
git push origin main

# 也合并到 dev
git checkout dev
git merge hotfix/fix-xxx
git push origin dev

# 删除 hotfix 分支
git branch -d hotfix/fix-xxx
```

## 四、运行机工作流

### 4.1 首次部署

```bash
git clone https://github.com/<your-username>/quantforge.git
cd quantforge
git checkout main

# 创建 tokens/ 目录和文件（见第六节）
# 安装依赖
pip install -r requirements.txt
```

### 4.2 日常更新

```bash
cd quantforge
git checkout main
git pull origin main

# 如果 requirements.txt 有变化
pip install -r requirements.txt
```

### 4.3 运行机只拉 main

**铁律：运行机永远只在 main 分支上操作，不修改代码。**

如果运行中发现 bug，在开发机上修复，推送到 main，运行机再 pull。

### 4.4 运行机上的本地文件

运行机上会产生一些本地文件（数据缓存、日志、持仓状态等），这些文件已在 `.gitignore` 中排除，不会被 git 追踪：

```
data/               ← 数据缓存
logs/               ← 日志
results/            ← 回测产物
*.json              ← 持仓状态
```

## 五、.gitignore 配置

### 当前完整 .gitignore

```gitignore
# === Python ===
__pycache__/
*.pyc
*.pyo
*.egg-info/
dist/
build/

# === 虚拟环境 ===
.venv/
*.pyc
__pycache__/

# === 密钥（真实文件不提交）===
tokens/小熊同学token.py
tokens/wechat_webhook.py
tokens/email_config.py

# === 数据与缓存 ===
data/*          # 数据缓存文件
!data/.gitkeep  # 保留目录结构
logs/*          # 日志文件
!logs/.gitkeep  # 保留目录结构

# === 回测产物 ===
results/*         # 回测结果
!results/.gitkeep

# === 持仓状态 ===
position/*.json
!position/position.json.example

# === 开发工具排除 ===
.trae/          # IDE 配置
.mpm-data/      # MPM 索引数据

# === 临时文件 ===
debug.log
```

### 关键说明

| 排除项 | 理由 |
|---|---|
| `tokens/*.py`（真实文件） | 含 API token、webhook key、邮箱授权码等密钥 |
| `data/*` | 数据缓存文件，体积大且可重新获取 |
| `logs/*` | 日志文件，运行时产生 |
| `results/*` | 回测结果，可重新生成 |
| `position/*.json` | 持仓数据，个人隐私 |

## 六、tokens/ 目录跨设备同步方案

由于 `tokens/` 不入版本控制，新设备需要手动创建。提供模板文件解决此问题。

### 6.1 模板文件

在项目中维护 `tokens/_templates/` 目录（入版本控制，只含模板不含真实值）：

```
tokens/
├── _templates/                    ← 入版本控制
│   ├── 小熊同学token.py.template
│   ├── wechat_webhook.py.template
│   └── email_config.py.template
├── 小熊同学token.py               ← 不入版本控制（真实值）
├── wechat_webhook.py              ← 不入版本控制（真实值）
└── email_config.py                ← 不入版本控制（真实值）
```

### 6.2 模板文件内容

`tokens/_templates/小熊同学token.py.template`：
```python
token = 'YOUR_API_TOKEN_HERE'
```

`tokens/_templates/wechat_webhook.py.template`：
```python
webhook_url = 'YOUR_WECHAT_WEBHOOK_URL_HERE'
```

`tokens/_templates/email_config.py.template`：
```python
sender = 'YOUR_EMAIL@163.com'
receiver = ['RECEIVER_EMAIL@qq.com']
authorization_code = 'YOUR_EMAIL_AUTH_CODE_HERE'
```

### 6.3 新设备部署步骤

```bash
# 1. clone 仓库
git clone https://github.com/<your-username>/quantforge.git
cd quantforge

# 2. 从模板创建 tokens 文件
cp tokens/_templates/小熊同学token.py.template tokens/小熊同学token.py
cp tokens/_templates/wechat_webhook.py.template tokens/wechat_webhook.py
cp tokens/_templates/email_config.py.template tokens/email_config.py

# 3. 填入真实值（用文本编辑器编辑上述文件）
notepad tokens/小熊同学token.py
notepad tokens/wechat_webhook.py
notepad tokens/email_config.py

# 4. 安装依赖
pip install -r requirements.txt

# 5. 验证
python -c "from tokens.小熊同学token import token; print(f'API token loaded: {token[:3]}***')"
```

### 6.4 .gitignore 配合

```gitignore
# tokens 目录下的真实文件不入版本控制
tokens/小熊同学token.py
tokens/wechat_webhook.py
tokens/email_config.py

# 模板文件入版本控制
# tokens/_templates/ 通过不匹配上述规则自然被追踪
```

## 七、AI 辅助开发时的 Git 协作

### 7.1 AI 提交规范

AI 辅助编码时，提交信息遵循 Conventional Commits 格式：

| 前缀 | 用途 | 示例 |
|---|---|---|
| `feat:` | 新功能 | `feat: 实现 RankingResolver` |
| `fix:` | 修复 bug | `fix: 修复 BacktestExecutor 资金计算错误` |
| `refactor:` | 重构 | `refactor: 提取 RiskGuard 链条` |
| `docs:` | 文档 | `docs: 更新策略配置说明` |
| `chore:` | 杂务 | `chore: 更新 .gitignore` |
| `test:` | 测试 | `test: 添加 ROC 策略回测验证` |

### 7.2 AI 编码后的 Git 操作

AI 完成编码后，需要：

1. **检查变更**：`git status` + `git diff --stat` 确认 AI 修改了什么
2. **分步提交**：不同逻辑的修改分开提交（代码修改 vs 文档 vs 配置）
3. **推送**：`git push origin dev`（需人工确认）
4. **运行机同步**：在运行机上 `git pull origin main`（如果已合并到 main）

### 7.3 AI 不应自动执行的操作

| 操作 | 原因 |
|---|---|
| `git push` | 需要人工确认后再推送 |
| `git merge` 到 main | 需要人工确认 Phase 完成 |
| 修改 `tokens/` 文件 | 密钥不应由 AI 修改 |
| 修改 `.gitignore` | 需要人工确认排除规则 |

### 7.4 文件删除策略（防阻塞）

`DeleteFile` 工具会触发 `requires_approval` 阻塞整个任务流。但文件删除几乎从不是其他步骤的前置依赖。

**规则：将删除操作推迟到最后执行，不要卡在中间。** 先做完所有无阻塞的读写工作，最后一次性批量删除需要清理的文件。

## 八、常见场景操作手册

### 8.1 开发机完成一个功能，运行机部署

```bash
# 开发机
git checkout dev
git add <精确文件列表>
git commit -m "feat: 实现 XXX"
git push origin dev

# 验证通过后，合并到 main
git checkout main
git merge dev
git push origin main

# 运行机
cd quantforge
git pull origin main
```

### 8.2 运行机发现问题，开发机修复

```bash
# 开发机（在 dev 分支修复）
git checkout dev
# ... 修复代码 ...
git add <精确文件列表>
git commit -m "fix: 修复 XXX 问题"
git push origin dev

# 合并到 main
git checkout main
git merge dev
git push origin main

# 运行机
git pull origin main
```

### 8.3 两台设备上 tokens/ 文件不同步

不需要同步——每台设备的 `tokens/` 是独立的，只要文件存在且值正确即可。

### 8.4 数据缓存文件的处理

`data/` 目录内容不入版本控制。两台设备各自维护数据缓存：
- 开发机：回测时自动缓存
- 运行机：实盘监控时自动缓存

如果运行机需要历史数据，首次运行时会自动从 API 拉取并缓存。

### 8.5 回测结果文件的处理

回测结果不入版本控制。如果需要跨设备查看：
- 方案A：在开发机上重新运行回测
- 方案B：手动拷贝结果文件（如通过微信/邮件发送）

## 九、开源安全注意事项

### 9.1 开源前的代码审计

每次推送到公开仓库前，必须确认：

1. 全文搜索无密钥/令牌/个人邮箱残留
2. `tokens/` 目录下只有 `_templates/` 和 `__init__.py`
3. `.gitignore` 正确排除所有敏感文件
4. 没有硬编码的绝对路径和 IP 地址

### 9.2 开源后的开发流程

- 开发在**私有仓库**（或本地）进行
- Phase 完成后，将代码同步到**公开仓库**
- 同步前执行上述审计流程

### 9.3 历史提交安全

不使用 `git filter-branch` 等历史重写工具（会影响协作者）。新仓库从 `git init` + 单 commit 开始，天然没有历史泄露风险。

cd quantforge

# 3. 创建 .gitignore（见第五节内容）
