# 任务驱动开发流程

> **自动触发**：当用户要求"规划"、"拆任务"、"施工文档"时，自动遵循本规范。

---

## 一、核心理念

**先规划 → 出施工文档 → 按序执行 → 对照验收**

```
Phase 整体规划讨论
       │
       ▼
  拆分子任务，确定执行顺序
       │
       ▼
  为每个子任务生成《施工文档》
  存放于 ToDo/ 目录，顺序编号
       │
       ▼
  按编号顺序逐个执行
  每完成一个，标记状态并验收
       │
       ▼
  全部完成 → 合并到主分支
```

---

## 二、施工文档模板

**存放位置**：`ToDo/` 目录

**命名规则**：`T{序号}_{简短描述}.md`，序号用三位数（如 `T001_ROC指标封装.md`）。序号即为执行顺序。

**必须包含以下章节**：

```markdown
# T{序号} {任务标题}

> **状态**：[pending | in_progress | completed]
> **所属阶段**：[Phase 1/2/3/4]
> **创建日期**：YYYY-MM-DD
> **完成日期**：（完成后填写）

## 一、前置依赖

- 依赖哪些已完成的任务（如：T001 完成后才能开始）

## 二、目标

- 清晰的一两句话，描述本任务要达成什么

## 三、实施计划

1. 具体步骤 1
2. 具体步骤 2
3. ...

## 四、验收标准

- [ ] 具体的、可验证的条件 1
- [ ] 具体的、可验证的条件 2
- [ ] ...

## 五、相关文件

| 文件 | 类型 | 说明 |
|------|------|------|
| `core/xxx.py` | 新建/修改 | ... |

## 六、备注

- 实施过程中的注意事项
- 可能的坑
```

---

## 三、执行纪律

1. **讨论阶段可以畅所欲言**，但不能直接写代码
2. **施工文档生成后**：按文档步骤逐步执行编码，不做文档之外的事
3. **一个任务未完成前**：不开始下一个，除非有明确理由需要调整顺序
4. **验收失败时**：分析原因，更新施工文档，重做
5. **任务定义有问题时**：回到规划阶段重新讨论，不自行修改任务范围
6. **完成后更新状态**：在文档开头标记 `completed`，填写完成日期

---

## 四、状态标记

| 状态 | 含义 |
|------|------|
| `pending` | 尚未开始 |
| `in_progress` | 正在执行 |
| `completed` | 已完成并验收通过 |
