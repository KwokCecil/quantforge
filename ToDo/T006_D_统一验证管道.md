## 一、前置依赖

- **T001 FactorLab**：验证管道的第 2-3 步需要 IC 分析
- **T002 Validator**：验证管道的第 4-5 步需要参数扫描和 Walk-Forward

## 二、目标

实现 `ValidationPipeline` 类，将零散的验证步骤串联为一个统一的、可配置的验证流水线。一个策略上线前，跑一条命令即可完成全部统计验证。

## 三、实施计划

### Step 1：设计 ValidationPipeline 接口

```python
# research/validation_pipeline.py

class ValidationPipeline:
    """统一验证管道。按顺序执行：数据质量 → 因子IC → 参数稳健性 → Walk-Forward → 成本建模"""

    def __init__(self, strategy_class, config_class, data_feed,
                 codes: list[str], start: str, end: str):
        ...

    def run(self,
            steps: list[str] = None,   # 默认全部，可指定子集
            param_ranges: dict = None,  # 参数扫描范围
            train_years: int = 3,
            test_years: int = 1) -> dict:
        """
        返回:
          {
            'data_quality': {...},
            'factor_ic': {...},
            'param_robustness': {...},
            'walk_forward': {...},
            'cost_sensitivity': {...},
            'verdict': 'PASS' | 'FAIL' | 'WEAK_PASS',
          }
        """

    def _step_data_quality(self) -> dict:
        """数据质量检查：缺失日期、停牌日(vol=0)、异常值(价格跳空>10%)"""

    def _step_factor_ic(self) -> dict:
        """因子有效性检验：IC/ICIR/分层单调性"""

    def _step_param_robustness(self, param_ranges) -> dict:
        """参数稳健性：最优参数 ±30% 范围内是否仍有效"""

    def _step_walk_forward(self, train_years, test_years, param_ranges) -> dict:
        """Walk-Forward 样本外验证"""

    def _step_cost_sensitivity(self) -> dict:
        """交易成本敏感度分析"""
```

### Step 2：实现 _step_data_quality()

数据质量是验证的第一步。基础检查：
- 日期连续性：报告缺失的交易日日期
- 停牌日标记：vol=0 的行统计
- 价格异常：单日涨跌 > 10% 的行
- 数据跨度：每个标的的最小/最大日期

输出为报告字典，不是阻断性的。质量差只警告，不阻止后续步骤。

### Step 3：实现 _step_factor_ic()

调用 T001 的 `FactorLab.compute_ic()` 和 `layered_backtest()`，自动对当前 `ROCConfig` 的默认参数（roc_n=22）做 IC 分析。

### Step 4：实现 _step_param_robustness()

- 以当前最优参数为中心，生成 ±30% 范围内的多组参数
- 调用 T002 的 `Validator.parameter_sweep()` 跑局部扫描
- 评估：最优参数附近是否有一个"稳健平原"（周围参数表现不急剧恶化）？

### Step 5：实现 _step_walk_forward()

直接调用 T002 的 `Validator.walk_forward()`。

### Step 6：实现 _step_cost_sensitivity()

- 测试不同佣金/滑点假设下的收益变化
- 默认测试：commission_rate = [0.0001, 0.00025, 0.0005]、slippage = [0.0005, 0.001, 0.002]
- 输出：成本每提高 1bp，收益率降低 X% 的敏感度

### Step 7：实现整体判定逻辑 (verdict)

```
PASS:      全部 5 步通过（ICIR>0.3, params稳健, WF正收益, 扣成本后盈利）
WEAK_PASS: 部分指标勉强达标，建议谨慎
FAIL:      关键指标不达标（ICIR<0.1, WF负收益），不建议实盘
```

判定逻辑可在 `ValidationPipeline` 中作为可配置的阈值，不硬编码。

### Step 8：创建 CLI 入口

```python

python main_validate.py --strategy roc_momentum
python main_validate.py --strategy roc_momentum --steps data_quality,factor_ic
```

## 七、备注

- **不要过早做判定系统**：verdict 的逻辑应该是"参考建议"，不是"自动挡"。目前阶段策略上线最终由你人工判断
- **数据质量检查的深度**：L3 阶段只做基础检查。停牌日的 ffill 偏差标记是重要的已知问题（规格书 §3.7），在质量报告中要标注
- **管道的结果可视化**：每个步骤输出的字典即可，不需要在这个阶段做图表。后续可以改进
- **与 T007/T008 的协同**：验证管道产出的报告，可以与 T008 的回测版本管理打通——验证报告和对应代码版本一起存档
