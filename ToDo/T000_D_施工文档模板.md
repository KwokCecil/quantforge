## 六、备注

- 状态字段七个可选值：
  - `pending`（待执行）
  - `in_progress`（执行中，修改核心文件，不可并行）
  - `parallel_safe`（执行中，仅操作独立文件，其他任务可同时进行）
  - `completed`（已完成）
  - `blocked`（被阻塞，等待依赖或人为决策）
  - `deprecated`（已废弃/被推翻/被替代，移入 archive 目录）
- 并行安全字段：
  - `是（纯新建文件）`：只创建新文件，不修改已有模块，可与其他 task 无条件并行
  - `是（修改独立文件）`：修改的文件与其他 task 不冲突，可与不冲突的 task 并行
  - `否（修改核心文件）`：修改的模块可能被其他 task 依赖或冲突，需要独占
  - `⚠️ 待定`：依赖前置验证结果
- 占用文件字段：列出本 task 将要**修改**的所有文件（新建文件不含冲突风险，可选择性列出）
  - 开启新 task 前，先扫描所有 `in_progress` / `parallel_safe` 任务的 `占用文件`
  - 如果新 task 的占用文件与任一进行中 task 的占用文件重合 → 不可并行，需等待
  - 如果无重合 → 可并行开工
- 序号用三位数，从 001 开始
- **文件命名规则**：`T{序号}_{状态字母}_{简短描述}.md`，状态字母直接标记在文件名中，一眼可见：
  - `P` = Pending（未开始，含 blocked）
  - `I` = In Progress（进行中，含 parallel_safe）
  - `D` = Done（已完成）
  - `X` = Deprecated（已废弃/被推翻/被替代，移入 ToDo/archive/ 目录）
- 状态变更时，同步修改文件名中的状态字母
- 一个任务标记 `in_progress` 时，确保没有其他任务也在 `in_progress`；但 `parallel_safe` 状态允许多个同时存在
- 验收标准要尽量具体、可验证，不要写"代码质量好"这种主观标准

### 并行工作区操作流程

```
开启新任务前:
  1. 检查所有进行中任务（in_progress / parallel_safe）的「占用文件」
  2. 对照自己计划修改的文件
  3. 无冲突 → 开工，标记状态（parallel_safe 或 in_progress），文件名状态字母改为 I
  4. 有冲突 → 等待，或在 TODO 中加备注说明阻塞原因

任务完成时:
  1. 更新状态为 completed
  2. 修改文件名状态字母为 D
  3. 填写完成日期
  4. 如有产出文件，更新「相关文件」表

合并顺序建议:
  1. 先合 pure parallel_safe 的（纯新建，零风险）
  2. 再合 in_progress 的（改核心，需要检查冲突）
  3. T013→T014→T015 必须按序合（有严格依赖链）
```
