### 3.3 缺口识别

按 3.10 文档"什么场景值得写单测"标准，以下模块处于 **裸奔** 状态：

| 模块 | 状态 | 风险 |
|------|------|------|
| BacktestExecutor 内部逻辑（非契约） | ⚠️ 无单测 | 改了买入算法/资金分配，契约对比看不出（都错但一致） |
| RankingResolver | ⚠️ 无单测 | 权重分配、止损、挤压卖出逻辑复杂，改任一参数都可能意外影响 |
| MacroOverlayResolver | ⚠️ 无单测 | CDR 分段映射 + 平滑 + 趋势过滤链式逻辑，手算验证困难 |
| TimingResolver | ⚠️ 无单测 | enter/exit 映射 + bond_etf 对端配置 + rebalance 阈值 |
| 通用技术指标函数（MA/REF/STD 等） | ⚠️ 无单测 | 被所有 Indicator 依赖的底层函数 |

## 四、实施计划

### 约定

- 所有单元测试使用 **合成数据**（`_make_bar_data` 模式），不依赖网络/文件/缓存
- 断言 **具体数值**（股数=cash=N 元），不测试模糊指标（"收益为正"）
- 运行时间 < 1s
- 文件命名 `tests/_test_{模块}_unit.py`
- 文件顶部标注 `# @layer: unit`

### Step 1：Executor 单元测试 — `tests/_test_executor_unit.py`

**目标**：覆盖 BacktestExecutor 内部所有路径 —— 买入、卖出、追加、整手取整、资金不足降级、调仓减仓、成本计算。不重复契约测试已经验证的 BE/LE 对齐。

#### 测试用例

```
test_buy_new_codes()
  输入: 空仓, 初始资金 40000, 5个标的等权(0.2), 价格 [1.5,2,3,4,5], stop_small_trade=False
  期望:
    - 每只标的买入股数 = floor(money / (price*1.001) / 100) * 100
    - cash > 0 (有找零)
    - 5个持仓全部建仓
    - total_commission > 0, total_slippage > 0

test_buy_full_capital()
  输入: 空仓, 初始资金 40000, 1个标的 weight=1.0, 价格 2.0
  期望:
    - shares = floor(40000 / (2*1.001) / 100) * 100
    - cash = 40000 - shares*2*1.001 - max(trade_value*0.00025, 0)
    - shares > 0
    - 1个持仓

test_buy_insufficient_cash()
  输入: 空仓, 初始资金 1000, 1个标的 weight=1.0, 价格 10.0 (最贵)
  期望:
    - shares = floor(1000/(10*1.001)/100)*100 = 0 (不足一手)
    - 无交易, 持仓为空

test_buy_insufficient_cash_adjust_down()
  输入: 空仓, 初始资金 3000, 1个标的 weight=1.0, 价格 10.0
  期望:
    - shares = floor(3000/(10*1.001)/100)*100 = 200
    - cash 约 3000 - 2000*1.001 - commission

test_sell_all()
  输入: 已持有 A(1000股, avg_cost=1.5), 价格 2.0
        targets = [TargetPosition("A", 0.0, "sell")]
  期望:
    - 全部卖出, A 不在持仓中
    - cash 增加 1000 * 2.0 * 0.999 - commission

test_sell_partial()
  输入: 已持有 A(1000股), 价格 2.0
        通过调仓逻辑: target_weight=0.5 (当前 weight≈1.0)
  期望:
    - 部分卖出, A 剩余股数 < 1000
    - A 仍在持仓中

test_average_cost_after_追加()
  输入: 已持有 A(1000股, avg_cost=1.5), 价格 2.0
        targets = [TargetPosition("A", 1.0, "追加")]
  期望:
    - 新 avg_cost = (1000*1.5 + new_shares*2.001) / (1000 + new_shares)
    - avg_cost < 2.001 且 > 1.5

test_high_watermark_update()
  输入: 已持有 A(1000股, hwm=1.5), 价格 2.0
  期望:
    - execute 后 high_watermark = 2.0

test_high_watermark_no_update()
  输入: 已持有 A(1000股, hwm=3.0), 价格 2.0
  期望:
    - execute 后 high_watermark 仍为 3.0

test_net_value_recorded()
  输入: 任意 execute
  期望:
    - len(net_values) > 0
    - 每条包含 date, net_value, cash, total_value

test_small_trade_skip()
  输入: stop_small_trade=True, skip_small_trade_limit=2000
        交易金额 < 2000
  期望:
    - 该笔买入被跳过, 持仓不变

test_rebalance_sell_excess()
  输入: 已持有 A(10000股, avg_cost=1.5), 总资产 50000, 价格 2.0
        targets = [TargetPosition("A", 0.3, "调仓")]
  期望:
    - 当前市值 20000, 目标市值 15000, 卖出 (20000-15000)/(2*0.999)/100*100 股
    - 持仓 A 仍存在
```

**预估工作量**：~200 行，12 个场景。验证方式：`python tests/_test_executor_unit.py` 全部 PASS。

test_top_k_sell_immediate()

test_squeeze_sell()

## 七、当前限制与说明

### 7.1 不测什么

- **策略类本身**（`ROCStrategy`、`BondYieldStrategy`、`MacroOverlayStrategy`）：这些类的核心逻辑是"接收 DataResponse → 产出 Decision"。对其写单元测试需要构建完整的 bar_data + macro_data，本质上就是集成测试。当前由 `_test_pipeline.py` 等集成测试覆盖。等 Executor/Resolver 单测稳定后，策略类的单元测试可以后续补充。
- **LiveExecutor 独立单测**：BE/LE 的对齐已由 `_test_executor_contract.py` 覆盖。LiveExecutor 剩下的路径（通知器、持久化、dry_run）依赖外部状态（文件系统、通知器对象），不适合纯单元测试。
- **StyleRotator**：风格轮动逻辑依赖真实大盘数据计算趋势，不属于单元测试范畴。

### 7.2 为什么现在达不到最佳效果

按 3.10 文档的理想目标，应该是"每次改 executor 或 resolver → 先跑对应的单测 + 契约测试"。当前的问题是：

1. **单测文件还没写**——**✅ 已解决。** 6 个文件全部创建并通过（57 个场景）。
2. **没有自动化运行机制**——**✅ 已解决。** `tests/run_all_tests.py` 已实现，glob 自动发现 + @layer 自动分层，`--all` 跑全部 14 个测试。
3. **覆盖率没有衡量标准**——不引入 pytest-cov 的前提下，只能靠"改了什么→跑了对应测试没有"的人工纪律。3.10 文档中的"提交前检查清单"是目前唯一的约束。

### 7.3 实施建议

- **Step 1 → Step 2 → Step 3/4/5 → Step 6** 按优先级顺序执行。每个 Step 完成后即可提交，不必等全部完成。
- Step 1 完成后先跑契约测试确保没破坏现有行为。
- 执行过程中如果发现被测模块的 bug，先写复现测试（失败），再修代码（通过），最后提交。
